Enterprise VLAN Segmentation Migration

Taking a flat homelab network from every device on a single 192.168.1.0/24 broadcast domain to a fully segmented 10-VLAN production design — with a hardware swap mid-migration that broke 25+ services, a Proxmox cluster quorum failure, and 30+ services needing IP and config updates before anything came back online.

OPNsense Cisco SG300 Netgear GS728TP 802.1Q VLANs Proxmox Firewall Design DNS Migration Troubleshooting

STAR Breakdown

  • Situation: The entire homelab ran on a flat 192.168.1.0/24 network — Proxmox nodes, game servers, SIEM, storage, IoT devices, and DMZ services all on the same broadcast domain with no traffic isolation. A hardware swap from legacy Cisco 3750V2 switches to a Cisco SG300-20 + Netgear GS728TP immediately took down 25+ services and left the Proxmox cluster without quorum.
  • Task: Design a production VLAN architecture, restore connectivity after the switch swap, migrate every service to its correct VLAN and IP, enforce per-VLAN firewall rules in OPNsense, and update all DNS, NAT, NPMplus, and application configs to point to the new IPs.
  • Action: Designed a 10-VLAN segmentation scheme, restored Proxmox corosync quorum via SG300 trunk configuration, migrated 30+ services with IP reservations, rewrote 25 DNS host overrides, updated 7 NAT port-forward rules, reconfigured 18 NPMplus proxy backends, migrated storage to VLAN 20, and deployed 80+ explicit OPNsense firewall rules.
  • Result: Fully segmented production network — 10 active VLANs, Proxmox cluster healthy (3/3 quorum), all services reachable, firewall default-deny enforced, and zero unnecessary cross-VLAN exposure.

Where It Started

When the homelab was first built, everything lived on 192.168.1.0/24. That's fine when you're getting started — one switch, one subnet, one gateway. But at some point the lab had grown to 22 VMs and LXC containers plus physical switches, NAS units, APs, and OOB management cards all on the same network. A guest device on the WiFi could technically reach the Proxmox management API. The Minecraft game server was on the same broadcast domain as the Wazuh SIEM and the Synology NAS. None of that is acceptable in a network that's supposed to model enterprise security practices.

The plan: redesign the network around 10 VLANs, each with a purpose, each enforced at the OPNsense firewall with explicit rules. The addressing scheme would follow 10.226.X.0/24 where X is the VLAN ID, with .1 as the gateway, .2–.49 static infrastructure, .50–.99 reserved static for services, and .100–.200 DHCP pool.

The VLAN Design

The 10-VLAN production layout, each with a specific security posture:

The key security decisions built into the design:

  • VLAN 2 (Game Servers) — internet yes, internal blocked. Pelican Panel needed to reach Pelican Wings on the same VLAN, but nothing in VLAN 2 should reach storage, SIEM, or Proxmox.
  • VLAN 3 (Servers) — Wazuh receives syslog and agent traffic from all other VLANs. Every VLAN has rules allowing outbound to the Wazuh manager IP on ports 1514/1515/514. VLAN 3 itself can't reach other VLANs beyond what's needed.
  • VLAN 4 (Testing) — fully isolated sandbox. Lab VMs and Kali instances get internet access but are hard-blocked from all production VLANs.
  • VLAN 5 (IoT) — internet only, RFC1918 blocked. Smart home devices and cameras cannot reach any infrastructure VLAN under any circumstance.
  • VLAN 6 (Services) — internal services gated by Authentik SSO. Jellyfin, Headscale, and AI inference run here. NPMplus in VLAN 7 is the only external entry point.
  • VLAN 7 (DMZ) — cloudflared and NPMplus only. cloudflared talks to NPMplus. NPMplus only reaches approved backend IPs and ports. Nothing in the DMZ can initiate to management.
  • VLAN 8 (Guest) — internet only, hard block on all RFC1918 destinations, captive portal via OPNsense.
  • VLAN 9 (OOB Mgmt) — IPMI, iDRAC, and IMM2 cards. Reachable from VLAN 10 only. No internet access, no lateral movement possible.
  • VLAN 10 (Management) — admin PC, OPNsense, Proxmox nodes, switches. The only VLAN allowed to initiate connections to everything else. No service VLANs can reach VLAN 10 unless explicitly allowed.
  • VLAN 20 (Storage) — no internet initiation. VLAN 10 and Proxmox nodes reach storage ports; storage devices get DNS and NTP but can't initiate back into the lab.

Phase 1 — The Switch Swap That Broke Everything

The legacy switching was a pair of Cisco Catalyst 3750V2 switches — 100 Mbps FastEthernet to everything. Both were replaced with a Cisco SG300-20 core switch and a Netgear GS728TP PoE access switch, both gigabit. The physical swap was quick. The aftermath wasn't.

The new switches came up with default configs. VLAN 50 (the legacy cloudflared DMZ) wasn't trunked anywhere. The Proxmox nodes were plugged into the SG300 but VLAN 10 wasn't configured on those ports yet. Every service that depended on the network came down immediately.

Restoring cloudflared first

The first priority was getting public ingress back. cloudflared was on VLAN 50 at 192.168.50.2 — but VLAN 50 wasn't tagged on any SG300 port. Fix: add VLAN 50 to the SG300 VLAN database, tag it on gi19 (OPNsense) and gi1 (Proxmox), confirm the LXC gets its DHCP lease, and verify the Cloudflare tunnel registers: 

! Cisco SG300 — restore VLAN 50 path for cloudflared
vlan database
 vlan 50
 exit
interface gi1
 switchport trunk allowed vlan add 50
interface gi19
 switchport trunk allowed vlan add 50
copy running-config startup-config

After applying: cloudflared at 192.168.50.2 could ping OPNsense at 192.168.50.1, the Cloudflare tunnel registered, and public services came back online.

Proxmox quorum failure

The bigger problem: Proxmox corosync had already been configured to use the VLAN 10 addresses (10.226.10.2/3/4 for the three Proxmox nodes) but VLAN 10 wasn't tagged on the SG300 ports facing nodes 2 and 3 — only gi1 (node 1) had it. Corosync uses multicast on a shared VLAN to maintain quorum. Without VLAN 10 reachability, the cluster showed 1/3 nodes and all cluster operations were blocked.

The fix required identifying exactly which physical switch ports connected to each Proxmox node using CDP/LLDP, then adding VLAN 10 tagged to those ports: 

! Confirm ports via LLDP
show lldp neighbors detail

! gi1 = pve-node-1, gi2 = pve-node-2, gi3 = pve-node-3
interface range gi1-3
 switchport trunk allowed vlan add 10
interface gi19
 switchport trunk allowed vlan add 10
copy running-config startup-config

! Verify quorum restored
pvecm status
# Quorum information
# Node votes: 3
# Expected votes: 3
# Total votes: 3
# Quorum: YES

Phase 2 — VLAN Cutover: 30+ Services

With the cluster healthy and public services restored on the legacy IPs, the actual VLAN migration could begin. Every service needed to move from its 192.168.1.x address to its new 10.226.X.Y address. This wasn't just changing IPs — every dependency had to be updated at the same time or services would break the moment the IP changed.

The dependency chain

For each service the update chain was:

  1. Set DHCP reservation in OPNsense with the new VLAN IP
  2. Update the LXC/VM network config in Proxmox (VLAN tag on the bridge interface)
  3. Reboot or reconfigure the container to pick up the new IP
  4. Update OPNsense Unbound DNS host override to point the subdomain to the new IP
  5. Update NPMplus proxy host backend to the new IP
  6. Update any service-to-service configs (Authentik URLs, Jellyfin NFS mounts, etc.)
  7. Verify via firewall logs that the rule allowing that traffic existed

Doing all 30+ services by hand through the OPNsense and Proxmox UIs would have taken days and been error-prone. Instead, the DNS overrides and NPMplus backends were updated in bulk via the OPNsense API: 

# Update all 25 Unbound DNS host overrides via OPNsense API
# Each subdomain (auth, jellyfin, status, opnsense, etc.) gets its new IP

curl -sk -u "$KEY:$SECRET" -X POST \
  https://[opnsense-mgmt-ip]/api/unbound/settings/addHostOverride \
  -H "Content-Type: application/json" \
  -d '{"host":{"enabled":"1","hostname":"auth","domain":"masternazz.com",
       "rr":"A","server":"10.226.6.50","description":"Authentik SSO"}}'

# Repeat for all 25 subdomains, then apply:
curl -sk -u "$KEY:$SECRET" -X POST \
  https://[opnsense-mgmt-ip]/api/unbound/service/reconfigure

Services that needed special handling

Most LXC containers were straightforward — change VLAN tag, reboot, pick up new IP, update DNS. A few required more work:

Authentik (VLAN 6, .50): OPNsense had Authentik configured as both an LDAP auth server and an OIDC provider. Both configs had hardcoded IPs. Updated via the diag_backup.php config restore mechanism (since direct writes to /conf/config.xml require root access that the API account doesn't have).

Jellyfin media stack (VLAN 6, .55): The NFS mount from Synology was still pointed at the old VLAN 1 IP. After Synology moved to VLAN 20, the NFS export had to be re-added, and the Proxmox storage definition updated. The Docker bind mounts inside the LXC then picked up the correct path.

Proxmox Backup Server (VLAN 20, .10): The Thecus NAS running PBS needed its network interface reconfigured from 192.168.1.xxx/24 to 10.226.20.10/24, the SG300 port set to access VLAN 20, and the PBS storage target in Proxmox updated to the new IP.

Wazuh SIEM (VLAN 3, .50): Every other VLAN needed firewall rules allowing outbound to Wazuh's IP on ports 1514 (agent), 1515 (registration), and 514 (syslog). These rules were added per-interface so each VLAN could reach Wazuh without having general inter-VLAN access.

Windows Server 2025 (VLAN 3, .120): Moved to VLAN 3 and the Wazuh agent was installed via WinRM to start forwarding Windows event logs into the SIEM.

Phase 3 — Firewall Rules

OPNsense uses a default-deny policy — if there's no rule allowing traffic, it's blocked. Every VLAN needed a minimal set of rules:

  • Allow DNS to the OPNsense gateway (AdGuard Home on the management interface)
  • Allow Wazuh agent/syslog traffic to the Wazuh manager IP in VLAN 3
  • Allow outbound internet where appropriate
  • Block RFC1918 destinations for untrusted VLANs (game, guest, IoT, testing)
  • Explicit allow rules for specific service-to-service paths

The most important rules are the inter-VLAN paths for DMZ egress. NPMplus in VLAN 7 needs to reach backends in VLAN 6, VLAN 2, and VLAN 10: 

! OPNsense firewall rules — VLAN 7 DMZ egress
# NPMplus -> VLAN 6 internal services (all backends)
pass  interface=VLAN7  src=10.226.7.51  dst=10.226.6.0/24  [NPMplus -> VLAN6]

# NPMplus -> VLAN 2 game server backends
pass  interface=VLAN7  src=10.226.7.51  dst=10.226.2.0/24  [NPMplus -> VLAN2]

# NPMplus -> VLAN 10 management backends (Proxmox, OPNsense)
pass  interface=VLAN7  src=10.226.7.51  dst=10.226.10.0/24 [NPMplus -> VLAN10]

# cloudflared can ONLY talk to NPMplus — nothing else in DMZ
pass  interface=VLAN7  src=10.226.7.50  dst=10.226.7.51    [cloudflared -> NPMplus]

The VLAN 7 implicit deny catches anything NPMplus tries to reach that wasn't explicitly whitelisted. If a new backend service is added, it needs a matching firewall rule before NPMplus can proxy to it.

Blockers Hit Along the Way

Proxmox routing conflict

After VLAN 10 was configured, some connections to legacy 192.168.1.x devices (Cisco switch, AP) started failing from Proxmox. Root cause: a stale manual IP in the 192.168.1.0/24 range had been added to vmbr0.10 (the VLAN 10 bridge interface) during earlier testing. This caused Proxmox to route all traffic destined for 192.168.1.0/24 out the VLAN 10 interface instead of the native bridge, where the physical legacy devices actually were. Fix: remove the stale address from the VLAN 10 interface.

Synology NAS — didn't survive port move

Moving the Synology NAS switch port from VLAN 1 to VLAN 20 required the DSM network settings to already be configured with the target VLAN 20 IP before the port move — otherwise the NAS goes silent the moment the port VLAN changes. The lesson: update the device IP first (while still reachable on VLAN 1), then move the switch port. The Synology migration required logging into DSM while still on VLAN 1, configuring the static IP for VLAN 20, and only then moving the SG300 port.

NPMplus backend updates

NPMplus stores proxy host configs as nginx conf files on disk. When every service IP changed, all 18+ proxy host backends needed updating. Rather than editing each one through the NPMplus UI, the nginx configs were patched via sed and nginx reloaded — but NPMplus regenerates configs on changes, so the correct approach was updating through the NPMplus API or UI so the database stayed in sync with disk.

VLAN 1 couldn't be retired immediately

Even after most services migrated, VLAN 1 stayed live because of out-of-band management hardware — the IBM M4 IMM2 (IPMI/remote console for firstmoon) was still on 192.168.1.x and needed physical access to reconfigure. Legacy VLAN 1 remains as a migration zone until OOB cards are moved to VLAN 10.

The End State

From start to finish this took roughly two weeks of iterative work across multiple sessions. The final production state:

  • 10 active VLANs, each on its own 10.226.X.0/24 subnet
  • Proxmox 3-node cluster healthy — 3/3 quorum on VLAN 10
  • All 22 VMs and LXCs assigned to their correct VLANs with production IPs
  • 25 DNS host overrides updated — all *.masternazz.com subdomains resolve to correct VLAN IPs
  • 7 NAT port-forward rules updated (MC server ports, Headscale STUN, TeamSpeak voice)
  • 18 NPMplus proxy backends updated to new IPs
  • 80+ OPNsense firewall rules — default-deny, explicit allows per path
  • Wazuh receiving logs from every VLAN via per-VLAN agent rules
  • Storage fully on VLAN 20 — Synology, QNAP, Thecus/PBS all isolated
  • Switching at 1 Gbps — SG300-20 core, Netgear GS728TP PoE access

Key Takeaways

Map dependencies before cutting over. Every service has at least 3–5 things that need updating when its IP changes: DHCP reservation, DNS, reverse proxy backend, any application configs that reference it by IP, and firewall rules. Missing one keeps the service broken even after the IP is right.

Bulk API updates beat the UI. Updating 25 DNS overrides and 18 proxy backends one at a time through a web UI takes hours and introduces typos. OPNsense and NPMplus both have APIs. Writing a script that applies all changes atomically and can be re-run is faster and leaves a record of what changed.

Hardware swaps during a migration are risky. Replacing the switches mid-migration added a full extra phase of work — re-trunking all VLANs, recovering quorum, restoring VLAN 50 for cloudflared. If the switch hardware had been swapped after the VLAN design was validated on the old hardware, the transition would have been cleaner. Order matters.

Device-by-device NAS migrations. Network devices that don't have a console (switches excepted) need their IP changed in software before the switch port moves. The Synology NAS taught this lesson the hard way — configure the target IP in DSM first, then move the port.

Default-deny firewall forces you to document traffic flows. Every path that needs to work has to become an explicit rule. That's annoying during setup but it means you always know exactly which services can talk to which — no accidental cross-VLAN bleed.

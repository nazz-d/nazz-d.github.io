Game Hosting & Discord Automation

Self-hosted Minecraft server infrastructure running on Pelican Panel and Docker inside a Proxmox LXC. Built a Discord bot from scratch that pulls live server stats via RCON and Docker metrics — deployed on its own dedicated LXC with a systemd service.

Pelican Panel Docker Python Discord API RCON Protocol Proxmox LXC systemd

Game Server Infrastructure

  • Deployed Pelican Panel (open-source successor to Pterodactyl) across two dedicated LXC containers — one for the panel, one for the Wings daemon.
  • Wings runs game servers as isolated Docker containers, giving each server its own resource limits, filesystem, and network namespace.
  • Minecraft server runs on Fabric 1.21 with custom mods (Spark performance profiling, etc.) inside the Docker container managed by Wings.
  • RCON enabled for remote command execution, used by the Discord bot to query live game state.
  • Port allocations managed through the Pelican admin panel — game port (25565) and RCON explicitly allocated and forwarded.

Discord Bot — Built From Scratch

  • Wrote a custom RCON client in pure Python implementing the Minecraft RCON wire protocol — no external RCON libraries.
  • Bot polls the server every 5 minutes and edits a single pinned embed (no channel spam) showing live player count, online players, in-game time of day, day number, difficulty, uptime, CPU, memory, and disk usage.
  • CPU / memory pulled from docker stats; disk from df inside the container; uptime from Docker's StartedAt timestamp — all via SSH chain through Proxmox.
  • Slash commands: /status (full embed on demand) and /players (quick player list).
  • Bot's own Discord status updates dynamically: green "Playing mc.masternazz.com • N/20 online" when up, red "Offline" when down.
  • Deployed on a dedicated Debian LXC as a systemd service — starts on boot, auto-restarts on failure.
Protocol

Custom RCON Client

The Minecraft RCON wire protocol is a binary TCP protocol. Rather than pulling in a library, I implemented it from scratch in Python — packet framing, struct packing, auth handshake, and response parsing. This lets the bot query any RCON command: player list, in-game time, difficulty, and day number.

Observability

Docker Metrics via SSH Chain

The bot LXC can't reach the Docker socket directly. Instead it SSH's into the Proxmox host, which uses pct exec to run docker stats and docker inspect inside the Wings LXC — pulling real CPU, memory, disk, and exact uptime for the MC container and surfacing them in Discord.

Deployment

Isolated LXC + systemd

The bot runs on its own Debian LXC (VMID 122) — not bundled into the Wings host or any other service container. A systemd unit file handles startup, restart-on-failure, and boot persistence. Python venv keeps dependencies isolated. The whole thing is self-contained and independent of the game server lifecycle.

How It All Connects

Discord Channel
    ↑  embed edit / slash command response
    │
mc-bot LXC
    ├─ RCON → Pelican Wings LXC
    │           └─ Docker container: Minecraft server
    │               ├─ player list, time, day, difficulty
    └─ SSH → firstmoon Proxmox
                └─ pct exec → Wings LXC
                    └─ docker stats / docker inspect
                        ├─ CPU, memory, disk
                        └─ exact container uptime

This isn't a "spin up a server and forget it" project. I built the monitoring layer myself — implemented a binary protocol, chained SSH through Proxmox to reach a Docker socket I didn't have direct access to, wrote async Python for the Discord API, and deployed it as a proper systemd service on dedicated infrastructure. It's the kind of pragmatic problem-solving and end-to-end ownership that infrastructure and DevOps roles expect.

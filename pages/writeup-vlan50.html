Cisco Trunking & VLAN 50 Fix

A deep dive into identifying and resolving a VLAN trunking mismatch between OPNsense and a two-switch Cisco Catalyst fabric to restore connectivity for a DMZ-isolated Cloudflare Tunnel.

Cisco IOS VLAN 50 802.1Q Trunking Troubleshooting OPNsense Cloudflare Tunnel

STAR Breakdown

  • Situation: The Cloudflare Tunnel LXC, isolated in a dedicated Public_DMZ (VLAN 50), was unreachable and failing to receive a DHCP lease from the OPNsense gateway.
  • Task: Verify the physical and logical path of VLAN 50 through the network fabric—consisting of OPNsense, a Proxmox cluster, a main Cisco 3750V2, and a secondary Cisco 3750V2.
  • Action: Systematically audited the switch configurations. Identified that while VLAN 50 was tagged at the OPNsense and Proxmox layers, the Cisco trunk ports connecting the switches were only allowing the native VLAN (VLAN 1). Applied switchport trunk allowed vlan add 50 across the entire path.
  • Result: Restored full L2/L3 pathing. Cloudflared received lease 192.168.50.2, successfully reached the gateway, and restored all global status and web services.

The Symptom

The cloudflared service is critical to my infrastructure—it's how my public status page and secure web ingress function without opening ports on my WAN. During an update, the container lost connectivity. nmap from the LAN couldn't reach it, and it was failing to get a DHCP lease on the 192.168.50.0/24 segment.

The Topology Audit

My network uses a "Router on a Stick" (mostly) model with multiple VLANs. The path for VLAN 50 is:

  1. OPNsense ( em0.50 ) -> Gateway
  2. Cisco 3750V2 Main ( Fa5/0/47 ) -> Trunk to OPNsense
  3. Cisco 3750V2 Main ( Fa5/0/48 ) -> Trunk to Sub Switch
  4. Cisco 3750V2 Sub ( Fa2/0/47 ) -> Trunk back to Main
  5. Cisco 3750V2 Main ( Fa5/0/1 ) -> Trunk to Proxmox Node

If any single link in this chain fails to "allow" VLAN 50, the L2 broadcast domain is broken and the DHCP Discover packet never reaches the OPNsense server.

Troubleshooting Steps

1. Verifying OPNsense & Proxmox

I first checked the OPNsense interfaces. VLAN 50 was active on em0, and the DHCP server was running. In Proxmox, the LXC was correctly tagged with bridge vmbr0 and tag 50. The issue wasn't at the endpoints.

2. The Cisco Discovery

I logged into the main switch (192.168.1.182) and ran show interfaces trunk. The results were clear: VLAN 50 was missing from the "Vlans allowed on trunk" list. The ports were trunking, but they were effectively filtering out the DMZ traffic.

3. The Implementation

I moved systematically through the fabric to apply the fix: 

! On Cisco_Catalyst_3750_V2_Main
interface range Fa5/0/1, Fa5/0/47, Fa5/0/48
 switchport trunk allowed vlan add 50
 exit
wr mem

! On Cisco_Catalyst_3750_V2_Sub
interface Fa2/0/47
 switchport trunk allowed vlan add 50
 exit
wr mem

The Result

Immediately after saving the configuration, the cloudflared LXC received a DHCP lease (192.168.50.2). I verified connectivity with a ping to the gateway (192.168.50.1) and confirmed that the Cloudflare Tunnel registered as "Active" in the Zero Trust dashboard.

Key Takeaways

Trust but Verify: Just because a port is set to mode trunk doesn't mean it's carrying your traffic.

Documentation is a Tool: Having a clear map of which physical ports connected my OPNsense box to my switches allowed me to identify the exact interfaces to audit in seconds rather than guessing.

DMZ Isolation: This fix was particularly important because it maintained the security boundary. VLAN 50 remains isolated from the LAN, but now it has the L2 path it needs to reach the internet.
